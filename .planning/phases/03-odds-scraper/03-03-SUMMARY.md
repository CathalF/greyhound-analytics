## Decisions Made

### Architecture Decisions

**Refresh interval: 5 minutes between cycles**
- Rationale: Balance between real-time updates and rate limit respect
- Implementation: `--interval` flag (default 5 minutes)
- Impact: Odds updated 12 times per hour, sufficient for value bet identification

**Lookahead window: 4 hours (default)**
- Rationale: Focus on imminent races, avoid wasting resources on distant races
- Implementation: `--hours` flag controls window
- Impact: Processes 50-100 races per cycle (manageable with rate limiting)
- Flexibility: Can be adjusted (2-8 hours) based on needs

**Status lifecycle: upcoming → imminent → complete**
- upcoming: Race more than 30 minutes away
- imminent: Race within 30 minutes (higher priority)
- complete: Race time passed (excluded from future refreshes)
- Rationale: Focus refresh effort on active races, stop updating completed races
- Impact: Prevents wasted work, improves efficiency

**Shared rate limiter: Single TokenBucketRateLimiter across all scrapes**
- Rationale: Global 2-second delays prevent site throttling
- Implementation: `rate_limiter = TokenBucketRateLimiter(rate=0.5)` shared by all OddscheckerScrapers
- Impact: Respects oddschecker rate limits, prevents blocking

**Error handling: Continue on failures**
- Rationale: One failed race shouldn't crash entire refresh cycle
- Implementation: try/except around each race with logging, continue to next
- Impact: Robust operation, failures logged for review

### Data Flow

**Race Discovery**:
1. Navigate to `https://www.oddschecker.com/greyhounds`
2. Parse all race links (pattern: `/greyhounds/{track}/{time}/winner`)
3. Filter out "Closed" races
4. Return list of upcoming race URLs

**Refresh Cycle**:
1. Discover races (504 found in test, filtered to ~50 within 2-hour window)
2. For each race:
   - Scrape race card (OddscheckerScraper)
   - Store race (upsert_race)
   - Determine status based on time until race
   - Match dogs to stats database (match_dog_to_stats)
   - Link matched dogs to race (link_dog_to_race)
   - Store odds for matched dogs (upsert_odds)
   - Log success/failure
3. Print summary statistics
4. Sleep for interval (if continuous mode)

### Logging Strategy

**Log Levels**:
- INFO: Progress (races discovered, processed, summary stats)
- DEBUG: Detailed info (dog matching, status changes, field extraction)
- WARNING: Dog match failures, minor issues
- ERROR: Scraping failures, database errors

**Log Outputs**:
- Console: INFO and above (for monitoring)
- File: DEBUG and above (for troubleshooting)
- File rotation: 10MB max, 5 backups

**Log Format**:
```
2026-01-14 16:04:47 - INFO - Discovering races for next 2 hours...
2026-01-14 16:05:13 - INFO - Found 504 upcoming races

```

## Issues Encountered

### Minor Issues (Resolved)

**Unicode encoding error in logging (Windows)**
- Issue: Unicode checkmark (✓) in logger.info() caused encoding errors on Windows console
- Impact: Logging errors displayed in output (but file logging still worked)
- Resolution: Replaced ✓ with [OK] in logger messages
- Result: Clean output, no more encoding errors

**High race count for broad time windows**
- Issue: 504 races discovered for 2-hour window (too many for single cycle)
- Impact: Would take ~4 hours to process all races
